# OSS 開発のコントリビュートに挑戦してみた

私自身はプログラミングを学びはじめて 4 ヶ月程度ですが、OSS gate のイベントに参加し、はじめて OSS に PR を出してみました！今回はその参加のレポートを書いてみます。

# OSS gate のイベント概要

> 「OSS Gate」とは OSS 開発に参加する人を増やす取り組みです。2015 年 12 月 15 日から始めた取り組みです。
> 取り組みの名前に「Gate」と入れたのは、まだ OSS 開発に参加していない人が参加する人に変わる「入り口」を提供する、という理由からです。

## OSS 開発について学ぶ

代表的なフレームワークや日常的に使っているツールはすべてオープンソースですが、それは当たり前ではなくて、多くのコントリビューターや公式のメンテナーによって継続的にメンテナンスされているのですよね。
自分で使うツールを自分たちで改善していく事ができる OSS 開発の文化を学びました。

また、具体的に以下のことを学びました

- OSS のライセンスの調べ方（OSS かどうかを確認できる）
- OSS の GitHub リポジトリへの Issue の作り方、PR の作り方（OSS 毎に異なるので確認をする）

## 開発したい OSS を GitHub などで検索

OSS のライセンスを確認するには対象リポジトリの`LICENSE`ファイルに記載があります。
以下の WEB サイトでオープンソースのライセンスなのかを確認できます。
@[card](https://opensource.org/license)

## OSS 開発をサポートしてくれるメンターとともに OSS の開発

1. 対象にした OSS：Fluentd
   Fluentd はログ収集ツールです。今回メンターをしてくださった[@daipom](https://x.com/daipompom)さんは Fluentd の公式メンテナーだったので、今回は Fluentd を対象にしました。

   @[card](https://www.fluentd.org/)

2. OSS のライセンスを確認
   `LICENSE`ファイルを確認し、`Apache License Version 2.0, January 2004 http://www.apache.org/licenses/`の記載がありました。

3. Fluentd を使ってみる
   Fluentd をそもそも触ったことがなかったので、まずはローカルで Fluentd を使ってみました。
   メンターと一緒に生成されるログファイルや`README.md`を確認しながら、改善できそうな部分を探していきました。具体的には、メンターと GitHub 上で作業メモを残しながら行いました。

そこで出てきたのは、公式ドキュメントの `Quick Start` のページが、生成されるファイルの拡張子がわかりにくいということに気が付きました。
公式ドキュメントの記述をはじめて fluentd を使う人にもわかりやすいように改善する、という方向性で進めることにしました。

4. 実際に OSS に PR を作成し変更を送るまで
   1. OSS のリポジトリを GitHub 上で`Create Fork`
      これで自分の GitHub 上でで OSS のコピーが管理ができるようになります。
   2. Fork したリポジトリを`git clone`
   3. ローカルで作業ブランチを作成し変更を加える
      今回は公式ドキュメントの Quick Start の説明文の改善なので`out-file-example`というブランチを作成し、公式 Docs に修正を行いました。
   4. ローカル PC から Fork されたリポジトリへ Commit を Push
   5. OSS 本家のリポジトリで Issue や PR を作成
   6. メンテナーが Issue や PR の変更を本家の OSS に取り込む

こちらの記事がとてもわかり易いです。

# OSS 開発を通して学んだこと

- OSS 開発は自分で使っているツールの不具合の修正や改善活動に貢献できる
- Issue や PR のコメントは誰もが読んでわかるような記述を意識する（エラーのソースコードや、やったこと、やっていないことを省略しないで記述）
- Issue の作り方や Pull Request 時の注意点などポイントを抑えることが、OSS 開発に関わる不安の解消方法

# 最後に

オープンソースというのは、ツールの開発者やメンテナンスをする人、そしてツールのユーザーが関わっているんだということを改めて認識しました。当然のように使っているフレームワークやライブラリも開発者の貢献があってこそ。今回のイベントで学んだことを土台にして、OSS への貢献にチャレンジしたいです。
